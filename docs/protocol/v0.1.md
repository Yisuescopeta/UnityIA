# Protocol v0.1

Every request returns an `ActionResult` with mandatory `success`, `message`,
`code`, and object-valued `data` fields. Exceptions never cross the public
boundary.

Mutation commands require:

- a valid session identifier;
- Edit Mode;
- the expected active scene;
- the current `contextVersion`;
- an explicit permission capability;
- explicit confirmation when the effective policy uses `confirm_actions`;
- a writable audit sink.

One request contains exactly one command. v0.1 has no command batches,
transactions, streaming, or nested commands.

Reusing a `commandId` in the same Editor session:

- returns the stored terminal result when the canonical payload matches;
- returns `IDEMPOTENCY_CONFLICT` when the payload differs.

`CONFIRMATION_REQUIRED` and `CONFIRMATION_DENIED` are pre-execution results.
They do not apply the mutation. After an approval for the same `commandId` and
canonical payload, the client retries the same command envelope.

The canonical schemas live in `schemas/`. The CLI validates against them.
Inside Unity, strict DTO deserialization and command-specific validators are
used instead of a JSON Schema engine.
